# Arcade Management System - dbms miniproject

This Arcade Management Web Application uses Spring Boot, standard JPA (Hibernate), Thymeleaf templates, and MySQL. It has been refactored to align closely with key software engineering principles. Furthermore, a user session authentication (login and database session storing via Spring Session JDBC) is active.

## Running the Application in WSL

In order to run this project properly within a WSL (Windows Subsystem for Linux) setup, ensure you have Java 17, Maven, and MySQL installed. Follow the steps below:

### 1. Database Setup
Start your MySQL service and log in:
```bash
sudo service mysql start
mysql -u root -p
```
Once inside MySQL, the database schema will be auto-generated by Hibernate during the Spring Boot run if you defined the properties. To make sure your connection works:
```bash
CREATE DATABASE IF NOT EXISTS arcade_db;
```
*(Optionally change user credentials in `src/main/resources/application.properties` to match your local setup if your root password isn't 'password')*

### 2. Build the Project
Navigate to the root directory where the `pom.xml` resides, and run:
```bash
mvn clean install
```

### 3. Run the Spring Boot Application
You can now spin up the server with:
```bash
mvn spring-boot:run
```

Once running, navigate to `http://localhost:8080/` via your Windows browser. You will be able to register a new player, log in, and see your HTTP interactions saved automatically in the database using Spring Session.

## Architectural Patterns

This project enforces structural and architectural robustness by adhering to:

### 4 GRASP (General Responsibility Assignment Software Patterns) Principles
1. **Information Expert:** We mapped behavior heavily to where the information resides. For instance, the exact logic on how player scores influence ranks (`checkAndUpdateRank()`) happens directly within `PlayerServiceImpl`, which possesses all the player score data.
2. **Creator:** Spring Data JPA repositories (such as `PlayerRepository` and `SessionRepository`) abstract the creation of Database records.
3. **Controller:** Incoming requests are isolated behind Spring MVC Controllers (`PlayerController`, `LoginController`). These classes act strictly as mediators—dealing with Web/HTTP semantics—and delegate all business logic to the underlying Service layer.
4. **Low Coupling & High Cohesion:** Responsibilities are physically and logically segregated (DTOs separate from Models from Web from Repositories). Interfaces are used abundantly (`PlayerService`) so that components communicate on abstractions, avoiding hard ties.

### 4 SOLID Principles
1. **Single Responsibility Principle (SRP):** Standard classes have exactly one responsibility. E.g., `GlobalExceptionHandler` enforces all error handling mapping; `LoginDTO` handles just data transportation for credentials.
2. **Open/Closed Principle (OCP):** Service structures rely on implementation layers (`PlayerService` interface vs `PlayerServiceImpl`). If a different database mapping is ever required, developers can introduce a new implementation without touching existing controllers extending from it.
3. **Liskov Substitution Principle (LSP):** Any object extending Spring's `JpaRepository` can be safely functionally substituted. Abstractions like `SessionService` don't leak logic, ensuring derived logic satisfies base contracts.
4. **Dependency Inversion Principle (DIP):** Every Controller invokes a Service *Interface*, not a raw class. Spring `@Autowired` injects the implementation dynamically, fully inverting the structural flow of control.
